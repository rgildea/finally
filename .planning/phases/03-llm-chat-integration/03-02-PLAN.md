<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → POST /api/chat | Untrusted user message enters the system |
| LLM → backend side-effect executor | LLM-generated trades/watchlist actions are untrusted and auto-executed |
| backend → SQLite | User/assistant content + actions JSON persisted |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-04 | Tampering | LLM-hallucinated trades (bad ticker, negative/oversized qty) | mitigate | Tickers normalized `.upper().strip()`; each trade goes through `execute_trade` which validates ticker/side/quantity/cash; ValueError caught and surfaced in reply, never crashes the endpoint |
| T-03-05 | Spoofing/Tampering | Prompt injection via req.message | mitigate | System prompt separates instructions from user input; LLM output re-validated by ChatResponse schema (Plan 01) before any side effect; only the typed trades/watchlist_changes are acted on |
| T-03-06 | Repudiation | Auto-executed trades have no record | mitigate | execute_trade appends to the immutable trades log (existing); assistant chat_messages row stores the actions JSON of what was executed |
| T-03-07 | Information Disclosure | Error strings leaking internals into the chat reply | mitigate | Surface only concise validation messages (e.g. "insufficient cash"); do not echo stack traces or the API key into resp.message |
| T-03-08 | Denial of Service | Synchronous blocking on litellm in the event loop | mitigate | get_llm_response (Plan 01) uses async `acompletion`; side effects use existing async price_cache + fast sqlite writes |
</threat_model>
